# 修复循环熔断机制

> QA Agent 追踪修复循环次数，触发阈值后自动熔断，避免无限打回-修复循环。

---

## 熔断规则

| 打回原因 | 最大自动修复轮次 | 超限行为 |
|---------|:---:|------|
| 需求不清晰（PO/BA 阶段） | 2 | 人工澄清 |
| 设计缺陷（Architect 阶段） | 2 | 人工决策 |
| 代码缺陷（Coder 阶段） | 2 | 熔断，人工 review |
| 测试不足（QA 阶段） | 2 | 熔断，人工验收 |

**全局上限**：全流程累计 ≤ 6 轮自动修复。超限后标记为 "需人工介入"。

---

## 熔断触发流程

```
QA 发现问题
  → 返回 Coder 修复（修复轮次 +1）
  → Coder 修复后重新提交
  → QA 重新验证
  → 如仍有问题：
    - 修复轮次 ≤ 2 → 返回 Coder（自动）
    - 修复轮次 > 2 → 触发熔断
```

---

## 熔断后行为

1. QA 报告标记 "⚠️ 已熔断"
2. 熔断原因和轮次记录到 qa-report.md
3. 暂停自动流程，通知用户
4. 用户决策：
   - 接受当前状态（Mark as done）
   - 手动修复后继续
   - 重新定义 AC（回退到 BA 阶段）

---

## 熔断状态记录

在 `qa-report.md` 中记录：

```markdown
### 修复循环

| 轮次 | 发现问题 | 修复状态 |
|:---:|---------|:---:|
| 1 | AC3 测试失败：登录后未跳转 | 已修复 ✓ |
| 2 | AC5 测试失败：权限检查遗漏 | 已修复 ✓ |
| 3 | AC7 测试失败：边界条件未处理 | 熔断 ⚠️ |

**熔断原因**：AC7 边界条件连续 2 次修复仍未通过
**建议**：人工 review AC7 的实现方案
```
