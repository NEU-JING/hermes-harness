# Brainstorming 指南

> Architect Agent 在 Design 阶段必须执行 Brainstorming（Standard/Enhanced 流程）。本指南定义方案对比的规范。

---

## 何时需要 Brainstorming

- Standard 流程：必须（R2 强制）
- Enhanced 流程：必须
- Quick 流程：跳过

---

## Brainstorming 步骤

### Step 1: 理解约束

列出技术约束：
- 现有技术栈（不能随意换）
- 团队熟悉度
- 性能要求
- 时间限制
- 基础设施限制

### Step 2: 生成候选方案

至少产出 **2 个**可行方案：

1. **保守方案**：基于现有技术栈，最小改动
2. **激进方案**：引入新技术/新架构，长期收益更大
3. **折中方案**（可选）：在两者之间平衡

### Step 3: 方案评估

对每个方案评估以下维度：

| 维度 | 权重 | 说明 |
|------|:---:|------|
| 实现复杂度 | 3 | 开发时间和难度 |
| 性能 | 2 | 响应时间、吞吐量 |
| 可维护性 | 3 | 代码可读性、模块化 |
| 可扩展性 | 2 | 未来扩展的容易程度 |
| 团队熟悉度 | 2 | 学习成本 |
| 风险 | 3 | 技术风险、依赖风险 |

### Step 4: 选择 + 记录

在 design.md 中记录：
- 每个方案的描述、优缺点、适用场景
- 对比矩阵
- 最终选择和理由

## 常见陷阱

1. **假对比**：方案 B 明显不可行 → 选出 2 个都可行的方案
2. **无数据支撑**："性能更好"无 benchmark → 提供具体数据或估算
3. **忽略团队约束**：选了一个团队完全没经验的方案 → 评估团队熟悉度
4. **跳过记录**：口头讨论但未写入 design.md → 必须记录到文档
