# 评审清单

> Reviewer Agent 的三阶段评审清单。

---

## Phase 1: Spec 合规

检查实现是否满足 Spec 中的功能需求。

- [ ] 所有 AC 被测试覆盖（逐个检查 AC 编号）
- [ ] 功能行为与 Spec 描述一致
- [ ] 未实现 Out of Scope 的功能
- [ ] 用户场景全部覆盖

**失败处理**：缺失 AC 覆盖 → 返回 Coder 补充

---

## Phase 2: 代码质量

检查代码是否符合通用质量标准。

- [ ] DRY：无重复代码
- [ ] YAGNI：无过度设计
- [ ] 函数 ≤ 50 行
- [ ] 命名清晰（不缩写）
- [ ] 错误处理完整
- [ ] 日志适当（不过多不过少）
- [ ] 无 TODO/FIXME（除非标注 issue）
- [ ] 前后端契约同步（R5，如有前端）

**失败处理**：质量不达标 → 标记严重级别，返回 Coder 修复

---

## Phase 3: 架构一致性

检查实现是否偏离 Design 文档。

- [ ] 模块划分与 Design 一致
- [ ] 接口定义与 Design 一致
- [ ] 数据流与 Design 一致
- [ ] 关键技术决策被遵循
- [ ] 无未声明的外部依赖

**失败处理**：偏离 Design → 判断是否为合理偏离
- 合理偏离（Design 未预见的边界情况）→ 记录到评审报告，标记为 INFO
- 不合理偏离（随意更改架构）→ 标记为 CRITICAL，需 Architect 确认

---

## 评审结论

| 结论 | 条件 |
|------|------|
| **通过** | 所有检查项通过，无 MAJOR 以上问题 |
| **有条件通过** | 无 CRITICAL 问题，MAJOR 问题有修复计划 |
| **不通过** | 存在 CRITICAL 问题，需打回 Coder |
