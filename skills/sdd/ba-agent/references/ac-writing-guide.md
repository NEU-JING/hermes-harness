# AC 编写指南

> Acceptance Criteria 是 SDD 流程的核心质量锚点。好的 AC 让 QA 阶段可自动化验证。

---

## AC 编号规范

- 格式：`AC{n}`，n 从 1 开始连续编号
- 禁止跳号（如 AC1 AC3 缺 AC2）
- 如删除某 AC，后续编号保持不变，在 Spec 中标注 "AC{n} 已删除"

## AC 格式

每条 AC 使用 **Given-When-Then** 三段式：

```
Given {前置条件}
When  {触发动作}
Then  {期望结果}
```

### 示例

**好的 AC**：

```
AC1: 用户登录
Given 用户已注册且未登录
When  用户输入正确的邮箱和密码并点击"登录"
Then  页面跳转到 Dashboard，显示用户名
```

**坏的 AC**：

```
AC1: 登录功能正常  ← 太模糊，无法验证
AC2: 用户应该能够登录系统  ← 缺少前置条件和期望结果
```

---

## 可测试性要求

每条 AC 必须满足以下条件之一：

1. **可自动化**：可通过单元测试或 E2E 测试验证
2. **可人工验证**：提供明确的验证步骤（仅限 UI 视觉效果）
3. **被标记为手动**：标注 `[MANUAL]` 前缀并说明原因

不可测试的 AC 示例（禁止）：

```
AC: 用户体验流畅  ← 主观，无法量化
AC: 系统稳定运行  ← 没有具体指标
```

---

## 可量化要求

每条 AC 的 Then 部分应包含可量化指标：

| AC 类型 | 量化维度 | 示例 |
|---------|---------|------|
| 性能 | 响应时间 | Then API 在 200ms 内返回 200 |
| 数据 | 条数限制 | Then 返回 ≤ 50 条记录 |
| 错误处理 | 状态码 | Then 返回 400 和错误消息 "邮箱格式无效" |
| UI | 元素存在 | Then 页面显示 "操作成功" 提示 |
| 权限 | 访问控制 | Then 非管理员访问 /admin 返回 403 |

---

## AC 数量建议

| 功能复杂度 | AC 建议数量 |
|-----------|:---:|
| Quick（小改动） | 1-3 |
| Standard（常规功能） | 5-15 |
| Enhanced（复杂功能） | 10-30 |

---

## AC 覆盖矩阵（QA 阶段使用）

QA Agent 会生成 AC → E2E 的覆盖矩阵：

| AC 编号 | E2E 用例 | 覆盖状态 | 备注 |
|:---:|------|:---:|------|
| AC1 | `login.spec.js: test_valid_credentials` | ✓ | |
| AC2 | `login.spec.js: test_invalid_email` | ✓ | |
| AC3 | — | ✗ | 无对应 E2E 用例 |

缺失覆盖的 AC 标记为 QA 不通过。
