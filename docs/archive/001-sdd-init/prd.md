# PRD: SDD 通用开发机制

> 版本：V1.0
> 日期：2026-05-24
> 作者：PO Agent
> 流程级别：Enhanced

---

## 背景

AILP 项目的「公开能力主页」试点验证了一套 SDD（Spec-Driven Development）流程：PRD → Spec → Design → Tasks → Implement → Review → QA → 验收。试点中产出了 7 个 AILP 专用的流程 Skills、6 条红线规则、4 套文档模板，但也暴露了系统性缺陷：

1. **流程依附于单个项目**：Skills 中硬编码了 AILP 的路径、技术栈、Constitution 引用，无法直接复用到其他项目
2. **角色定义分散**：PM/BA/Architect/Coder/Reviewer/QA 的职责、输入输出、禁止行为散落在多个文件中，Agent 执行时要么全量加载（浪费 token），要么凭记忆（遗漏步骤）
3. **缺少启动和约束机制**：新项目不知如何接入，存量项目升级风险高；Agent 执行时产物目录容易失控

经过两轮方案讨论，确定了"通用化 + 角色 Skill 化 + 项目配置化"的架构方向。本 PRD 定义落地范围。

## 目标

1. 构建一套**跨项目可复用**的 SDD Skills，任意项目通过一份精简的 AGENTS.md 即可接入
2. 将 7 个角色（PO/BA/Architect/Coder/Reviewer/QA/编排器）定义为独立 Skills，每个 Skill 是一份"角色说明书"——明确读什么、做什么、产出什么、交给谁、禁止什么
3. 提供项目启动机制（新项目一键初始化 + 存量项目安全升级）和合规约束（目录结构、产物完整性在阶段交接前自动验证）

## 目标用户

| 用户角色 | 特征 | 核心诉求 |
|---------|------|---------|
| 新项目开发者 | 空目录起步，想用 SDD 管理开发流程 | 一键初始化，自动生成骨架，立刻开始第一个变更 |
| 存量项目维护者 | 已有代码和文档，想升级到 SDD 流程 | 安全迁移，现有文件不丢失，计划透明可控 |
| SDD 编排器（Agent） | 按阶段调度角色，管理门禁和交接 | 每次交接前自动验证目录和产物完整性 |
| 各角色 Agent（PO/BA 等） | 接收委派任务，按规范产出文档或代码 | 任务指令清晰，知道读什么文件、用什么工具、输出到哪 |

## 核心功能点

### F1: 项目启动与升级

开发者触发初始化后，系统识别项目状态（新项目 / 存量项目），自动完成骨架搭建或升级。

- **新项目**：交互式询问项目信息（名称、技术栈、流程级别偏好），自动生成 AGENTS.md、CONSTITUTION.md、QUIRKS.md、目录骨架
- **存量项目**：分析现有文件，生成迁移计划（移动、新建、冲突标记），用户确认后执行，不覆盖任何用户修改过的文件
- 支持多种技术栈预设

### F2: 流程编排

编排器是 SDD 流程的唯一状态管理者，负责判定流程级别、调度角色、管理门禁、执行交接协议。

- 根据任务特征自动判定流程级别（Quick / Standard / Enhanced）
- 按 SDD 阶段顺序调度对应角色 Skill
- 每个阶段交接前强制调用合规验证
- 管理门禁：
  - **PRD 确认**：PRD 产出后，必须用户确认方向
  - **Spec 确认**：Spec 产出后，必须用户确认行为契约
  - **QA 熔断**：QA 失败 ≤4 轮自动返回 Coder 修复，第 5 轮仍失败则熔断等待用户决策
  - **人工验收**：QA 通过后，用户人工确认场景实现正确性。如有问题，按类型打回：
    - 需求理解偏差 → 回退到 PRD 阶段
    - 设计方案缺陷 → 回退到 Design 阶段
    - 代码实现 Bug → 回退到 Implement 阶段（对应 Task）
  - **归档条件**：人工验收通过 + CI 全绿 → 归档
- 归档时自动将变更目录从 changes/ 移至 archive/，更新 current/

### F3: 角色 Skill 体系

7 个角色各为一个独立 Skill，采用文档地图模式（核心指令 ~80 行 + references 按需加载）。

每个角色 Skill 明确定义：
- **触发条件**：什么情况下被调度
- **前置依赖**：需要读取哪些上游产物
- **执行流程**：分步指令，标注每一步"始终加载"还是"按需加载"
- **产出物**：输出什么文件、写入什么路径
- **交接清单**：向下游传递什么
- **禁止行为**：什么绝对不能做
- **退出条件**：什么状态下可以交接

角色列表：

| 角色 | 核心职责 |
|------|---------|
| PO Agent | 输出 PRD：Why + What + 用户故事 + 成功指标 |
| BA Agent | 输出 Spec：Given-When-Then AC + 业务规则 + 错误文案 |
| Architect Agent | 输出 Design + Tasks：Brainstorming 对比方案 + 技术方案 + 按业务场景拆分 Task |
| Coder Agent | 输出代码 + 测试：TDD 逐 Task 实现 + Task Completion Report |
| Reviewer Agent | 输出评审报告：自动化检查 + 独立审查 + 回归验证 |
| QA Agent | 输出 QA 报告：触发 CI 执行集成测试和 E2E 测试 + AC 覆盖矩阵验证 + 环境差异标记 |

### F4: 合规验证

在 SDD 流程的关键节点自动执行目录和产物完整性检查，不合规时阻断并给出修复指引。

- **项目级**：AGENTS.md / CONSTITUTION.md / 必要目录是否存在
- **变更级**：change-id 格式是否符合约定、产物文件是否存在于正确路径
- **交接级**：前置阶段产物是否完整、文件是否非空

### F5: 项目配置化

将原来 117 行的 AGENTS.md（混合了流程描述和项目配置）精简为约 50 行的纯配置文件。

- 流程描述从 AGENTS.md 移除，全部由 Skills 承载
- AGENTS.md 只声明：项目名、技术栈、路径约定、SDD 引擎引用、宪法和陷阱文件路径
- 支持 `convention_overrides`：项目可通过 AGENTS.md 覆盖默认约定（如"禁止按技术层次拆分 Task"）

## 不做的事（Non-Goals）

- 不开发 CLI 命令或 GUI 界面（SDD 通过 Hermes Agent 的 skill 系统调用）
- 不管理代码仓库（Git init、remote 配置属于开发者偏好）
- 不生成 CI/CD 配置（但保留与 CI 的衔接点，如 pre-commit hooks 模板）
- 不校验文档内容质量（由 Spec Linter 等后续 Skill 负责）
- 不包含项目特有的 Linter 规则（如 AILP 的 Spec Linter）
- 不提供可视化界面（编排器、流程状态均为命令行/消息交互）

## 成功指标

| 指标 | 目标值 | 衡量方式 |
|------|:---:|------|
| 新项目初始化耗时 | ≤ 3 分钟 | 空目录执行初始化，计时到"SDD 就绪" |
| 存量升级中断率 | 0% | git diff 验证现有文件未被修改 |
| 跨项目通用性 | 2/2 | AILP（FastAPI）+ mock 项目（Go）分别验证 |
| 角色 Skill 核心指令行数 | ≤ 80 行/Skill | 统计每个 SKILL.md 的正文行数（不含 frontmatter） |
| 编排器完整调度 | 8/8 阶段 | 在验证项目中走完整 SDD 流程，每个阶段产出符合约定 |
| 合规检查覆盖率 | ≥ 12 项 | 项目级 ≥ 5 + 变更级 ≥ 4 + 交接级 ≥ 3 |
